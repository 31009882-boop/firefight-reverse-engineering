[2026-04-03 10:10 | World instance model update | status taxonomy]
Current refined understanding:
1. `world.dat` instance data should no longer be described as a single yes/no success case.
2. The current reconstruction evidence supports at least four practical map families:
   - `canonical-0xCD00`
   - `canonical+unresolved-tail`
   - `variant-headers`
   - `sentinel-coordinates`
3. There is also a separate failure family:
   - `rejected-probe`
   This means the current assumption "prototype table end is immediately followed by an `Int32 declaredCount` for 8-byte instance records" is wrong on those maps, or at least incomplete.

What is currently strongest:
1. Stable maps such as `TRCH` still strongly support this record boundary:
   - `Int32 declaredCount`
   - then `count` records of 8 bytes
   - interpreted as `UInt16 headerWord`, `UInt16 x`, `UInt16 y`, `UInt16 prototypeIndex`
2. Mixed-header maps such as `BPAL` still preserve coherent coordinates and believable prototype hotspots.
3. This means the leading `headerWord` is increasingly likely to carry real per-instance semantics, not just a constant map-wide marker.

Working interpretation of the current map families:
1. `canonical-0xCD00`
   - likely the cleanest form of the instance family
   - best reference maps: `TRCH`, `TKYO`, `NSVL`
2. `variant-headers`
   - still plausibly the same 8-byte family
   - `headerWord` changes by instance and correlates with object clusters
   - best reference map: `BPAL`
3. `sentinel-coordinates`
   - records are still mostly spatially meaningful, but a minority of entries expose `0` / `65532` / slight overflow values
   - possibilities:
     - reserved markers inside the same table
     - disabled or hidden instances
     - alternate sub-record semantics
     - a second family partially sharing the same 8-byte footprint
4. `rejected-probe`
   - maps such as `JLRD`, `SELI`, and `UA1T` do not expose a plausible count word right after the current prototype-table boundary
   - this strongly suggests at least one extra block, alternate count encoding, or alternate instance-table family

Engine-recreation implication:
1. The future desktop runtime cannot hard-code a single universal `world instance` decoder.
2. The engine recreation will likely need:
   - a canonical instance decoder
   - one or more variant-header decoders
   - explicit handling for sentinel/reserved entries
   - a map-family detection step or a more faithful upstream block parser
3. Therefore, reverse-engineering the world layer now means recovering a dispatch model, not merely naming four fields.

Current best next-step lens:
1. Treat `TRCH` as the canonical baseline.
2. Treat `BPAL` as the mixed-header baseline.
3. Treat `CAMB` or `NSPN` as the sentinel-coordinate baseline.
4. Treat `JLRD` as the rejected-probe baseline.
5. The goal of the next round is to determine whether `headerWord` is:
   - flags
   - layer/class bits
   - orientation/state bits
   - or evidence of multiple interleaved record families.

[2026-04-04 14:46 | World instance model update | supplemental prototype tails]
Current refined understanding:
1. Some maps do not stop at a single main `world prototype table`.
2. After the final valid 8-byte world-instance record, there can be a second, map-local supplemental prototype segment.

   - re-use the same parsed instance records
   - extend prototype resolution with the post-instance supplemental name segment

   - but a nearby candidate is already parse-accepted as `sentinel-coordinates`
   - coordinate validity is still strong enough to show the structure is spatially coherent
   - current best example: `UA1T`
3. `garbage-offset-signal`
   - usually small positive nearby offsets such as `+9 / +17 / +25 / +33 / +41 / +49`
   - these produce `invalid-coordinates` explosions
   - header distribution becomes obviously fragmented / low-trust
   - this is now a strong negative signal rather than a candidate rescue path

Implication for next parser work:
1. `SELI` should now be treated as a prototype-resolution problem first, not mainly an offset problem.
2. `UA1T` should now be treated as a sentinel-adjacent / alternate-family problem first, not a clean-variant rescue that merely needs a small offset shift.
3. Future parser changes should target these families separately, instead of widening one universal offset rule.
