[2026-04-03 17:14 | Firefight 反编译工程长期记忆工作流 | 用户正式要求固化版]

用途：
1. 本文件是当前 `交战反编译项目` 的长期执行基线。
2. 每次新对话开始时，必须先读取本文件，再读取最新的 `游戏理解.txt / 计划任务备份.txt / 过往更新记录.txt / 过往对话记录.txt`，然后才开始实施。
3. 本文件优先服务当前项目：`Firefight` 安卓版反编译、逆向验证、桌面重建、最终完整可玩的 `.exe` 复刻。

一、项目总目标与停止条件
1. 项目最终目标不是查看器、报告器、半成品工具链，而是：
- 完整可玩
- 工程质量高
- 性能与结构经过优化
- 对原作运行时行为有足够逆向覆盖
  的桌面 `.exe` 版本。
2. 任何 `Workbench`、`Desktop`、报告、快照、可视化界面，都只属于中间层成果，不得误报为最终完成。
3. 若历史文档、旧计划、旧长期记忆与当前项目冲突，以：
- 当前源码真实行为
- 最新 `游戏理解.txt`
- 最新有效 handoff 文档
  为准。
4. 明显不属于当前项目的旧污染内容不得继承，例如和 Unity/PPO/训练体系有关、但与当前 `Firefight` 反编译工程不一致的条目。

二、每轮对话开始前的强制顺序
1. 先完整理解用户本轮原话，不得只抓关键词，要提炼：
- 本轮明确目标
- 隐含约束
- 必须更新的文档
- 必须推进的工程主线
- 用户新增的长期规则
2. 先检查工作区实际状态，尤其是：
- 上轮是否中断
- 源码与文档时间戳是否一致
- 是否存在未回填 handoff 的较新代码/报告
- 当前产物是否仍能构建
3. 在真正修改任何文件前，必须形成“本轮详细工作流/计划”。
4. 把这份详细计划先写入 `计划任务备份.txt`，要求下一位 AI 中断接手时能直接看懂：
- 本轮目标
- 计划步骤
- 预期修改文件
- 风险点
- 验证方式
- 未完成项和续作入口

三、subagent 协作规则
1. 每次对话默认组织多路 subagent 协作，不得只靠主线程单线推进。
2. subagent 要按任务分轨组织，常见分轨包括：
- 文档/规则校验
- `world.dat` 协议逆向
- `scenarios.dat` 协议逆向
- Desktop/Workbench 接线与可视化
- 验证/报告差异分析
3. 允许在过程中根据任务新形态继续生成新的 subagent。
4. 已完成任务的 subagent 要及时关闭，避免悬挂和上下文污染。
5. 主线程职责固定为：
- 需求收口
- 状态审计
- 决策优先级
- 最终代码修改
- 构建验证
- 文档回填
- 统一交接
6. 若当前会话工具不支持显式创建 subagent，也必须按 subagent 职能拆成并行分析轨执行，并在 `计划任务备份.txt` 里写清哪些轨道由主线程代执行。
7. 若某轮确实不适合再拆分，也必须在 `计划任务备份.txt` 写明原因；但默认倾向于并行分流。

四、备份与防污染规则
1. 任何修改前都必须先备份，不能先改后补。
2. 备份范围不只限于文档；凡是本轮要改的源码、配置、导出模板、报告入口、脚本，都要先备份。
3. 备份统一放在：
- `D:\桌面\交战反编译项目\交战反编译成果\备份`
  下的同一轮次临时目录中。
4. 每轮至少保留：
- 修改前临时备份
- 验证通过后的稳定快照
- 必要时刷新 `文档备份`
5. 只有在确认源文件无污染、修改结果验证无误后，才允许清理临时备份；稳定快照不能随意删。
6. 若发现文档里有跨项目污染、旧规则残留、编码错乱、手工复制误差，必须显式标注并在新一轮记录中说明如何处理。

五、文档回填规则
1. `计划任务备份.txt`
- 每轮开始前先写详细计划
- 每轮结束后补写结果、未完成项、下一位 AI 接手入口
- 内容必须足够具体，不能只写一句“继续推进”
2. `过往更新记录.txt`
- 记录本轮实际改了什么
- 结构要清楚，至少包含：类型、主要改动、影响文件、验证结果、备注
- 让接手者能快速知道项目已经做过什么
3. `过往对话记录.txt`
- 记录用户本轮要求、我本轮如何响应、得到的新结论、下一轮优先入口
- 重点写“用户意图”和“我实际做了什么”，不要只写流水账
4. `游戏理解.txt`
- 只写当前已稳定的理解、强证据推断、项目红线
- 不把暂时猜测包装成最终结论
5. 以上文档每次更新都必须考虑“中断接手可读性”。

六、拉尔夫循环式自检优化规则
1. 每轮实施必须执行“先做一版 -> 自检 -> 修问题 -> 再自检 -> 再优化”的循环。
2. 不接受“能跑就停”的低质量完成态。
3. 自检重点包括：
- 逻辑是否真实推进了反编译主线
- 是否只是 UI 装饰而没有证据增量
- 是否引入脆弱实现、过度猜名、误导性文案
- 是否让下一轮更容易继续推进
4. 若发现实现只是演示态、临时态、脆弱态，要继续优化到足够稳再收口。

七、工程推进主线
1. 当前优先主线仍是：
- `world.dat`
- `scenarios.dat`
- definition/availability
- tutorial special/script route
- world object instance
- 这些数据与未来 runtime 模型之间的桥接
2. 当前 `Workbench` 是逆向工作台，`Desktop` 是桌面可视化验证层。
3. 真正进入完整 runtime 之前，必须尽量把：
- world prototype
- world instance
- scenario deployment
- tutorial/script route
- definition/chassis/weapon/crew identity
  的关系链拆稳。

八、验证与收口规则
1. 每轮默认至少做与本轮相关的构建验证。
2. 若改到 `FirefightRebuild.Workbench` 或 `FirefightDesktop`，默认至少执行对应 `dotnet build -c Release`。
3. 若改到发布链、导出链、报告链，应尽量做最小 smoke 验证并落盘。
4. 收口时必须明确：
- 已完成
- 未完成
- 当前事实边界
- 下一轮最值得继续的点

九、编码与格式约束
1. 文档和文本输出保持 UTF-8，避免乱码回归。
2. 不得把推测硬命名成事实；对未确认字段用中性名称保真保存。
3. 不得把当前中间层成果误描述为“最终完整 Firefight”。
4. 任何新规则若和当前项目实际情况冲突，必须在新一轮计划/更新里说明取舍。

十、本文件的使用方式
1. 每轮开头先遵守本文件。
2. 每轮若用户新增了长期规则，要把它们并入本文件。
3. 若发现本文件本身需要优化，也要先备份，再更新，再验证文档无污染。
